# Persona: The Lead Dev

> "I need my AI tools to work *with* my workflow, not replace it with a black box I can't trust."

## Profile

| Field | Value |
|-------|-------|
| **Name** | "Alex" — The Lead Dev |
| **Age** | 28-42 |
| **Role** | Senior developer / Engineering lead at a startup or mid-size company |
| **Team size** | 3-15 engineers |
| **Experience** | 5-15 years shipping production software |
| **Tools today** | Claude Code, Cursor, GitHub Copilot, GitHub Actions, Linear/Jira |
| **Spending** | $20-100/month on AI coding tools personally; influences team tooling budget |
| **Tech comfort** | Very high — lives in terminal, reads source code of tools they adopt |

## Behaviors

- Evaluates new tools by installing them in a side project first, then proposing to the team
- Reads changelogs and GitHub issues before upgrading dependencies
- Values convention over configuration — wants opinionated defaults they can override
- Spends 30-40% of time on code review, architecture decisions, and mentoring
- Frustrated by tools that require constant babysitting or produce inconsistent results
- Will abandon a tool within a week if setup is painful or output is unreliable
- Prefers CLI-first tools; skeptical of heavy GUIs that hide what's happening

## Value Proposition Canvas

### Customer Jobs

| Type | Job |
|------|-----|
| Functional | Ship features faster without sacrificing code quality |
| Functional | Set up consistent development workflows across the team |
| Functional | Review AI-generated code efficiently and catch issues before merge |
| Functional | Maintain architectural consistency as the codebase grows |
| Social | Be seen as the person who makes the team more productive |
| Emotional | Feel confident that automated changes won't break production |
| Emotional | Reduce the cognitive load of context-switching between tasks |

### Pains

| Severity | Pain |
|----------|------|
| Critical | AI agents go off the rails — run wrong commands, get stuck in loops, require constant babysitting |
| Critical | "Almost right" code that takes longer to debug than writing from scratch (cited by 66% of devs) |
| High | Each AI tool is isolated — no orchestration between architecture, implementation, and review |
| High | Setting up AI workflows per-repo is manual, repetitive, and undocumented |
| High | Context window limitations mean agents lose track of large codebases |
| Medium | Trust erosion — 46% of developers actively distrust AI accuracy |
| Medium | No institutional memory — agents forget learnings between sessions |
| Low | Cost unpredictability when using AI agents at scale |

### Gains

| Impact | Gain |
|--------|------|
| High | A pipeline that handles the full cycle: design → implement → review → ship |
| High | Agents that understand *this specific* codebase's conventions, not generic patterns |
| High | Confidence that AI changes pass the same CI checks humans must pass |
| Medium | Parallel feature development without merge conflicts |
| Medium | Reduced context-switching — describe what you want, review what you get |
| Medium | Team-wide consistency — every developer uses the same automated workflow |
| Low | Agents that learn from past mistakes and don't repeat them |
| Low | Easy setup that works in any repo regardless of tech stack |

## Key Insight

> The gap isn't in AI coding capability — it's in **orchestration**. Developers have powerful individual tools (Cursor, Claude Code, Copilot) but no structured pipeline that coordinates them through a full feature lifecycle with project-specific context. They want a "team of agents" that knows their codebase, not a single chatbot that forgets everything between sessions.
