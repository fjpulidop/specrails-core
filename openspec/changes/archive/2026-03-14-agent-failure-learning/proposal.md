# Proposal: Agent Post-Mortem & Failure Learning Loop

## Problem

The current specrails pipeline has no memory of failure. When a reviewer agent catches a CI failure — a lint error, a broken test, a shell script quoting bug — it fixes it, logs it in the review report, and moves on. The next implementation cycle starts from zero. The same class of mistake recurs, the reviewer catches it again, and the fix cycle repeats indefinitely.

This is an institutional knowledge gap baked into the architecture. Individual agents accumulate memory (reviewer has `common-fixes.md`, developer has `failure-patterns.md`), but those are maintained by the agents themselves through organic learning. There is no formal write path from "reviewer found a failure" to "developer is warned about that failure class before starting work." The loop is broken.

The consequences are measurable:
- **Repeated fix cycles**: The reviewer spends fix attempts on the same classes of error across unrelated features.
- **No preventive prompting**: The developer has no explicit guardrail for patterns that have failed CI in this repo before.
- **Cross-agent knowledge decay**: Patterns the reviewer has learned are not automatically surfaced to the developer, test-writer, or future reviewers.

## Solution

When the reviewer agent detects a failure (test failure, lint error, shell script error, security scan issue), it writes a structured failure record to `.claude/agent-memory/failures/`. Each record is a small JSON file capturing what went wrong, why it went wrong, what file pattern it occurred in, and what prevention rule would have avoided it.

At the start of each developer agent run, the agent reads the most recent failure records that match the layer or file patterns it is working on, and includes those as explicit guardrails in its implementation approach.

Over time, `.claude/agent-memory/failures/` becomes the repo's institutional memory of "what goes wrong here" — inherited automatically by every future agent invocation.

The design uses the existing memory infrastructure (agent-memory directories, markdown + JSON files) and requires no new runtime dependencies. Both the reviewer and developer agent templates are the primary touch points.

## Scope

**In scope:**
- Failure record schema (JSON): `agent`, `error_type`, `root_cause`, `file_pattern`, `prevention_rule`, `timestamp`, `feature`
- Shared failure store at `.claude/agent-memory/failures/`
- Reviewer template (`templates/agents/reviewer.md`): add "Write Failure Records" section
- Developer template (`templates/agents/developer.md`): add "Read Recent Failures" step in Phase 1 (Understand)
- Generated instances: `.claude/agents/reviewer.md` and `.claude/agents/developer.md`
- Initial empty failures store: `.claude/agent-memory/failures/README.md`

**Out of scope:**
- Failure deduplication or aggregation logic (left for Phase 2)
- Automatic failure record expiry or archival (left for Phase 2)
- Failures surfaced to the architect or test-writer agent (left for Phase 2)
- A UI or dashboard for browsing failure records (Issue #1 — Agent Memory Inspector)
- CI integration for automatic failure record creation (no CI yet)
- Failure records generated by agents other than the reviewer

## Non-goals

- The developer does NOT automatically fix or suppress failure patterns. It reads them as guardrails — context that sharpens judgment, not a blocklist.
- The reviewer does NOT read existing failure records before reviewing. Its mandate is to catch what's there; the developer is the preventive layer.
- Failure records are NOT commit-time artifacts. They live in `.claude/agent-memory/failures/` which is in `.gitignore` scope by convention (agent-memory is local to the repo instance).

## Acceptance Criteria

1. `templates/agents/reviewer.md` includes a "Write Failure Records" section instructing the reviewer to write one JSON record per failure category found.
2. `.claude/agents/reviewer.md` reflects the same changes with placeholders resolved.
3. `templates/agents/developer.md` includes a "Read Recent Failures" step in Phase 1 (Understand), instructing the developer to read records from `.claude/agent-memory/failures/` that match the current feature's file patterns.
4. `.claude/agents/developer.md` reflects the same changes with placeholders resolved.
5. `.claude/agent-memory/failures/README.md` exists and documents the JSON schema.
6. The failure record JSON schema is consistent across the reviewer template, the developer template, and the README.
7. No new shell scripts, binaries, or external tool dependencies are introduced.
8. The change is fully backward-compatible — repos with no failure records yet behave identically to today.
