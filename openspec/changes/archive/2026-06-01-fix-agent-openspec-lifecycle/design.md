## Context

The three core agent templates (sr-architect, sr-developer, sr-reviewer) currently describe an OpenSpec lifecycle but do not structurally enforce it. Each agent hand-authors its own artifacts, simulates task completion, and defers archive to an unspecified future step. This means the OpenSpec system's lifecycle commands (`/opsx:new`, `/opsx:ff`, `/opsx:apply`, `/opsx:archive`) are never actually invoked by the agents, making the lifecycle aspirational rather than enforced.

Current failure modes:
- tasks.md is hand-authored by the architect rather than generated by `opsx:ff`; checkboxes are literary props, not actionable state
- The developer has no gate that prevents handoff when tasks are incomplete
- The reviewer archives without reading tasks.md — incomplete work can be archived
- The architect's frontmatter description triggers on `/opsx:ff`, causing the agent to call itself (circular self-trigger)

This is a meta-tool change: templates live in `templates/agents/` and are installed into target repos via `specrails-core init`/`update`. Changes are not retroactive to already-installed repos.

## Goals / Non-Goals

**Goals:**
- Make each agent structurally invoke the correct OpenSpec slash command at the correct lifecycle stage
- Add enforcement gates at each handoff boundary (architect→developer, developer→reviewer, reviewer→archive)
- Remove the circular self-trigger from sr-architect's frontmatter
- Propagate `specName` explicitly through all three agents as a required positional argument
- Mirror changes to installed agent files so specrails-core develops itself correctly

**Non-Goals:**
- Implementing or modifying the opsx slash commands themselves
- Changing any other agent (sr-product-manager, layer reviewers)
- Making changes retroactive to already-installed repos
- Changing how the orchestrator launches these agents (out of scope)
- Modifying CI configuration or the test matrix

## Decisions

**Decision 1: Agents invoke slash commands via the Skill tool, not Bash**

Slash commands like `/opsx:apply` are Claude Code skills, not shell executables. The templates must instruct agents to call them as `Skill("opsx:apply", specName)`, not as a Bash command. Ambiguous phrasing ("run the command") would be interpreted differently by different model instances.

Alternatives considered:
- Document as Bash invocation: rejected — slash commands are not in PATH; this would produce a tool error
- Leave invocation mechanism unspecified: rejected — ambiguity causes agents to guess, which breaks the lifecycle contract

**Decision 2: specName is a required positional argument; agents halt if absent**

The spec name must be passed when launching each agent. If not provided, the agent must halt with a clear error rather than guessing or deriving from context. Guessing would silently operate on the wrong change directory.

Alternatives considered:
- Derive specName from most-recently-modified change directory: rejected — fragile in multi-change workspaces; fails when two changes are active
- Make specName optional with fallback: rejected — the invariant "specName must be passed" is more important than convenience

**Decision 3: Checkbox gate uses exact Markdown syntax `- [ ]`**

The verification gate pattern must be explicit. The canonical unchecked checkbox in GitHub Flavored Markdown is `- [ ]` (hyphen, space, open bracket, space, close bracket). The template must specify this exact pattern so agents and test assertions match.

Alternatives considered:
- Use a regex like `/\[[ ]\]/`: too abstract for a prompt instruction; agents should have a concrete string to search for
- Check for absence of `[x]`: insufficient — a tasks.md with no checkboxes at all would pass this check incorrectly

**Decision 4: Template changes + installed file changes in same task sequence**

The installed files (`.claude/agents/`) must be updated in the same change as the templates. If only templates are updated, specrails-core will use the old agent behavior for its own development while all installed repos get the new behavior — an inconsistent state that would be confusing to diagnose.

**Decision 5: Remove circular self-trigger by updating frontmatter description, not by removing the agent**

The sr-architect is correctly triggered by the orchestrator with a specName argument. The problem is the description says it triggers on `/opsx:ff` (a user command), which causes re-entrancy. The fix is surgical: update the description to say the agent is launched by the orchestrator/user with a specName argument, removing the command-trigger language.

## Risks / Trade-offs

**[Risk] Orchestrator does not pass specName** → Mitigation: Agent halts with a clear error message ("specName is required — invoke this agent with the change name as argument"). The orchestrator contract is documented in this design; implementation must verify the orchestrator passes specName.

**[Risk] opsx:apply exits with error mid-task** → Mitigation: Developer agent's gate checks all tasks are `[x]` after apply exits. If apply exits with an error, tasks will still have `[ ]` items, and the gate will block handoff. The developer should re-invoke apply or report the failure.

**[Risk] Reviewer reads tasks.md before developer's apply fully updates it** → Mitigation: The handoff is sequential: developer completes Phase 3 gate (all [x]) before signaling the reviewer. The reviewer reads tasks.md only after receiving the handoff. No concurrent access issue.

**[Risk] Meta-tool regression: template rendering leaves `{{PLACEHOLDER}}` tokens in installed files** → Mitigation: The existing `reserved-paths.test.ts` and the new `agent-lifecycle.test.ts` provide regression coverage. The enrich step substitutes placeholders at install time; the new structural sections do not use any new placeholder tokens.

**[Risk] Installed file drift from template** → Mitigation: Task 7 explicitly mirrors changes. Future template regression is caught by the lifecycle invariant tests.

## Migration Plan

1. Update template files (Tasks 1–6) — no user-facing impact until `specrails-core update` is run
2. Update installed agent files for specrails-core itself (Task 7) — immediate effect on this repo's own development
3. Run `npm test` to verify zero regressions (Task 9)
4. On next `specrails-core init` or `update` in a target repo, the updated templates are installed

No rollback strategy needed: changes are additive behavioral corrections to prompt text. Git history preserves the prior state.
