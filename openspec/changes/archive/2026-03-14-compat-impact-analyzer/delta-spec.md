# Delta Spec: Backwards Compatibility Impact Analyzer

This document describes what changes in the spec system as a result of this OpenSpec change. It is the authoritative reference for what "before" and "after" look like at the spec level.

---

## New Specs

### `openspec/specs/compat-check.md` (NEW)

A new spec document describing the `/compat-check` command's behavior, flags, surface extraction rules, diff algorithm, migration guide format, and snapshot schema.

This spec follows the same structure as `openspec/specs/implement.md` and `openspec/specs/batch-implement.md`.

---

## Modified Specs

### `templates/agents/architect.md` (MODIFIED)

**What changes:** A new Phase 6 section is appended between the existing task breakdown guidance and the quality assurance section.

**Before:** The architect agent has five responsibilities: Analyze Spec Changes, Design Implementation Approach, Organize Tasks, Respect the Architecture, and Key Warnings.

**After:** The architect agent has six responsibilities. The new sixth responsibility is "Run Compatibility Check" — a mandatory phase that extracts proposed surface changes from the implementation design, compares them against the current API surface, classifies any breaking changes using the four-category taxonomy, and appends either a "Compatibility Impact" section (with migration guide) or a "Compatibility Notes" section (advisory) or a one-line no-change acknowledgement to the architect's output.

**Backwards compatibility of this change:** The architect agent's existing behavior is additive — the new phase appends to output rather than replacing any existing output. Existing OpenSpec workflows continue to work. The change is ADVISORY (Category 4: behavioral addition).

## New Templates

### `templates/commands/compat-check.md` (NEW)

A new slash command template. After `/setup` runs in a target repo, this is copied to `.claude/commands/compat-check.md`.

**Frontmatter schema:**
```yaml
---
name: "Compatibility Impact Analyzer"
description: "Analyze the current API surface for breaking changes..."
category: Workflow
tags: [workflow, compatibility, breaking-changes, migration]
---
```

**Placeholders used:**
| Placeholder | Description |
|-------------|-------------|
| `{{PROJECT_NAME}}` | Target repo project name (same as all other commands) |

No new placeholders are introduced. The command reads the filesystem directly rather than relying on project-specific configuration.

---

## New Generated Files (post-/setup)

### `.claude/commands/compat-check.md` (NEW)

Generated by `/setup` from `templates/commands/compat-check.md`. Created in the target repo after the installer runs.

### `.claude/compat-snapshots/` (NEW directory convention)

A local artifact directory created on first run of `/compat-check`. Stores surface snapshots as JSON. Not committed to VCS.

---

## install.sh Changes

**None.** The installer copies command templates as-is. The new `compat-check.md` template will be picked up automatically by the existing copy loop that processes `templates/commands/*.md`.

Verify this by checking install.sh's template copy logic handles the glob correctly — it should since it processes all `*.md` files in the directory.

---

## No-Change Areas

The following are explicitly NOT changed by this spec:

- `openspec/config.yaml` — no new config keys
- `templates/agents/developer.md` — developer agent unchanged
- `templates/agents/reviewer.md` — reviewer agent unchanged
- `templates/agents/product-manager.md` — product manager agent unchanged
- `install.sh` flags — no new flags
- Existing command templates — none modified
- `commands/setup.md` — setup flow unchanged; new command template is picked up by existing glob

## Surface Impact of This Change (Self-Referential)

This change introduces new contract elements to specrails' own API surface:

| Surface | Change | Category |
|---------|--------|----------|
| Command names | `/compat-check` added | Addition (non-breaking) |
| Command flags | `--diff`, `--snapshot`, `--since`, `--propose`, `--dry-run` added | Addition (non-breaking) |
| Template placeholders | No new placeholders | No change |
| Installer flags | None | No change |
| Architect agent output | New "Compatibility Impact" / "Compatibility Notes" section added | Category 4 Advisory |

**Verdict:** This change is fully backwards-compatible. No migration guide required.
