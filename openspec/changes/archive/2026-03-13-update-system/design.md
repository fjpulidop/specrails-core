## Context

specrails installs a complete agent workflow system into target repos via `install.sh` + `/setup`. The installer copies templates and the setup wizard adapts them (agents, personas, rules) to the project's stack. Currently there is no versioning, no update path, and no way to know what version a project has installed. As specrails adds features (web manager, new agents, improved templates), existing users are stuck on their initial install.

Key constraint: adapted artifacts (agents, personas, rules) were generated by analyzing the user's codebase and may have been manually edited. These cannot be blindly overwritten.

## Goals / Non-Goals

**Goals:**
- Users can update specrails in one command (`update.sh`) without losing customizations
- Version tracking from day one (install stamps version + manifest)
- Legacy installations (pre-versioning) are auto-migrated without reinstall
- Granular control: update everything or specific components via `--only`
- Safe by default: backup before changes, preserve on failure

**Non-Goals:**
- Automatic updates (no cron, no self-updating)
- Rollback command (backup folder is sufficient for manual rollback)
- Per-file cherry-picking within a component (too complex, low value)
- Changelog display during update (future enhancement)

## Decisions

### D1: Dual versioning — VERSION file + manifest checksums

- `VERSION` file at specrails root: human-readable, used in git tags, displayed in headers
- `.specrails-manifest.json` in target repo: checksums of every installed template file
- `.specrails-version` in target repo: plain text version string

**Why both?** VERSION + git tags is standard OSS practice for releases. Manifest checksums are the internal mechanism to detect what changed between versions without needing the specrails git history at update time. Alternative considered: git-diff-only approach — rejected because it requires the full specrails repo history and is fragile across clones.

### D2: Artifact categorization — three update strategies

| Strategy | Artifacts | Behavior |
|----------|-----------|----------|
| **Overwrite** | commands/, skills/, setup-templates/, web-manager/ | Replace silently. These are not project-adapted. |
| **Merge** | settings.json, security-exemptions.yaml | Add new keys/rules, preserve existing user values. |
| **Regenerate** | agents/, rules/, personas/ | Only if template checksums changed. Requires user consent. Runs `/setup --update`. |

**Why not overwrite agents?** They're generated by `/setup` analyzing the codebase — they contain project-specific conventions, stack details, and CI commands. Users may also have edited them manually.

### D3: Legacy migration — auto-generate manifest as v0.1.0

When `update.sh` finds `.claude/agents/` but no `.specrails-version`:
1. Assume v0.1.0 (only pre-versioning release)
2. Generate manifest by checksumming current specrails templates (not the user's adapted files)
3. Write `.specrails-version` as `0.1.0`
4. Continue with normal update flow

**Why checksum current templates?** The user installed from some point-in-time copy of specrails. We don't know exactly which, but v0.1.0 templates are the closest match. If templates diverged, the update will detect "changed" and offer to regenerate — which is the safe default.

### D4: Backup strategy — .claude.specrails.backup/

Before any modifications:
1. `cp -r .claude/ .claude.specrails.backup/`
2. Perform all updates
3. On success: `rm -rf .claude.specrails.backup/`
4. On failure (set -e trap): print message pointing to backup, do NOT delete

**Why folder copy over git stash?** The .claude/ directory may not be tracked in git (many users gitignore it). A folder copy works regardless of git state.

### D5: /setup --update — surgical regeneration mode

When `/setup` receives `--update`:
1. Read `.specrails-manifest.json` to identify which agent templates changed
2. Quick codebase re-analysis (same as Phase 1 of regular /setup, but skip user prompts)
3. Regenerate only the agents whose templates changed, using new templates + fresh codebase analysis
4. For new agent templates not previously installed: evaluate relevance against detected stack, ask user
5. Skip personas — they're user-defined, not template-driven (except The Maintainer for OSS)
6. Update any workflow commands that reference agents (e.g., /implement)

### D6: --only flag — component-based selective updates

| Component | What it updates |
|-----------|-----------------|
| `web-manager` | Only .claude/web-manager/ |
| `commands` | commands/ + skills/ |
| `agents` | Force agent regeneration (triggers /setup --update) |
| `core` | commands + skills + setup-templates (no agents) |
| `all` | Everything (default when no --only) |

## Risks / Trade-offs

- **[Risk] Manifest checksums diverge from actual installed files** — User manually edited a "core" file that we'd normally overwrite. → Mitigation: Overwrite is the documented behavior for core files. Users who customize commands should know they'll be replaced on update.
- **[Risk] Agent regeneration produces worse results than originals** — New template + re-analysis could generate inferior agents. → Mitigation: Backup exists. User can skip regeneration. Warning makes the trade-off clear.
- **[Risk] Large backup directory** — .claude/ with web-manager node_modules could be large. → Mitigation: Exclude node_modules from backup (`cp -r` with exclude pattern or rsync).
- **[Trade-off] Merge strategy for settings.json is additive only** — We add new keys but never remove deprecated ones. Keeps it safe but could accumulate cruft over time.
